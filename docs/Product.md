### 6.1 Portfolio View ✅

**주요 지표**

- 글래스모피즘 카드
- 수익/손실 동적 색상 (emerald / rose)

### 6.2 Strategy Management ✅

#### 퀀트 전략이란?
사람의 감이나 뉴스가 아닌, **데이터와 수식 기반으로 매매 판단을 내리는 규칙 시스템**.

예시:
- "주가가 20일 평균보다 5% 오르면 매수, 3% 떨어지면 매도"
- "PER이 낮고 최근 3개월 수익률이 높은 상위 20개 종목만 보유"

#### 핵심 구조: 규칙과 전략

```
규칙 (Rule)
개별 조건 하나
예: "20일 이동평균 돌파"
예: "거래량 급등"
예: "PER 15 이하"

전략 (Strategy)
규칙들을 묶은 단위
├── 필수 규칙 (AND): 모두 충족해야 신호 발생
└── 옵션 규칙: 충족하면 좋지만 없어도 신호 발생 가능
```

**전략 단위 설정**
- 폴링 주기: 전략마다 다르게 설정 (실시간 또는 특정 주기)
- 활성/비활성: 전략 단위로 관리

#### 화면 구조

```
전략 목록 → 전략 작성 → 전략 조합
```

---

#### ① 전략 목록

| 항목 | 내용 |
|------|------|
| 전략명 | 사용자 정의 이름 |
| 상태 | 초안 / 테스트 / 운영 / 중지 |
| 적용 시장 | KR / US |
| 최근 실행 결과 | 마지막 신호 발생 여부 및 시각 |

#### ② 전략 작성

**전략 작성 흐름**

```
[텍스트 모드 — 기본, 필수]
규칙명 입력 + 조건 텍스트 입력
예: "20일 이동평균을 돌파하면 매수"
        ↓ AI가 Python 코드로 자동 변환
        ↓ 저장

[코드 보기 — 선택, 등록 후 제공]
전략 상세 페이지에서 "코드 보기" 버튼
Python 코드 확인 및 직접 수정 가능 (Monaco Editor)
→ 코드를 아는 사용자만 사용하는 영역

[개별 테스트]
전략 하나씩 백테스트 수행
```

**설계 원칙**
- 코드를 몰라도 전략 작성 가능
- 코드를 아는 사용자는 등록 후 직접 수정 가능
- 코드 작성은 자유, 실행 구조는 표준화

**전략 코드 표준 인터페이스**

```python
def initialize(context):
    # 파라미터, 유니버스 설정
    pass

def generate_signal(data, portfolio, params, context):
    # data: 종목별 OHLCV 데이터
    # portfolio: 현재 포트폴리오 상태
    # params: 사용자 설정 파라미터
    # context: {"timeframe": "1d", "lookback": 60}
    return [
        {
            "action": "BUY",       # BUY / SELL / HOLD
            "symbol": "AAPL",
            "quantity": 10,
            "order_type": "MARKET",
            "reason": "20일 이동평균 돌파"
        }
    ]
```

**전략 유형**

| 유형 | 예시 |
|------|------|
| 가격 기반 | 이동평균, 볼린저밴드, 모멘텀 |
| 가치 기반 | PER, PBR, 배당수익률 |
| 거래량 기반 | 거래량 급등, 공매도 비율 |
| 복합 조건 | 위 유형들의 AND/OR 조합 |

**실행 트리거 (전략마다 설정)**

| 유형 | 예시 |
|------|------|
| 시간 기반 | 매일 장 시작 전, 매주 월요일 |
| 이벤트 기반 | 가격이 특정 조건 도달 시 |
| 수동 | 사용자가 직접 실행 버튼 클릭 |

---

#### ③ 전략 조합 (핵심 차별점)

전략을 개별로 만들고, **여러 전략을 AND 조건으로 묶어서** 모두 만족했을 때만 매매가 실행되는 구조.

```
전략 A: 20일 이동평균 돌파  →  ✅ 신호 발생
전략 B: PER 15 이하         →  ✅ 신호 발생
전략 C: 거래량 급등         →  ❌ 신호 없음

AND 조건: A ∧ B ∧ C  →  ❌ 미실행
```

**조합 흐름**

```
만들어둔 전략들을 AND 조건으로 묶기
        ↓
조합 전체 테스트 (최종 확인)
        ↓
활성화 → 실행
```

**조합 기능**
- 전략을 레고처럼 조합 가능
- 조합 내 전략별 신호 현황 모니터링
- 조건 충족 시에만 매매 실행
- 개별 테스트 후 → 조합 전체 테스트 순서로 진행

### 6.3 Execution Engine ✅ (API 키 연동 제외)

**목적**
- 전략을 실행하고 신호를 생성
- 신호 발생 시 알림 전송

**시세 API**
- 한국투자증권 OpenAPI (국내 + 미국 통합)
- 라이브러리: pykis

**트래킹 대상**
- 전략 안에서 유니버스 정의 (전략이 감시할 종목을 스스로 결정)
- 사용자가 별도 지정한 관심 종목

**폴링 방식**
- FastAPI BackgroundTasks 기반
- 폴링 주기는 전략 단위로 설정 (실시간 또는 특정 주기)

**신호 발생 시 알림 (카카오 나에게 보내기)**
- 종목명 및 신호 (BUY / SELL)
- 현재가
- 충족된 규칙 목록 (어떤 필수/옵션 규칙이 충족됐는지)

**실행 모드**

| 모드 | 설명 |
|------|------|
| 백테스트 | 과거 데이터로 전략 검증 |
| 페이퍼 트레이딩 | 실제 체결 없이 모의 실행 |
| 실거래 알림 | 신호 발생 시 알림, 실제 주문은 사용자가 직접 |

**실행 흐름**

```
한국투자증권 API로 실시간 시세 수신
      ↓
전략별 폴링 주기에 따라 규칙 체크
      ↓
필수 규칙 모두 충족 시 신호 발생
      ↓
카카오 나에게 보내기로 알림 전송
      ↓
사용자가 직접 증권사 앱에서 매매
```

**미결 사항 (4단계 Auto Trading 때 결정)**
- 주문 수량 결정 방식 (고정 수량 vs 포트폴리오 비중 기반)

## 7. 핵심 데이터 구조

| 개념 | 설명 |
|------|------|
| **Trade** | 거래 원장. 불변 기록 (symbol, side, quantity, price, datetime, source) |
| **Position** | Trade로부터 계산된 현재 보유 상태 |
| **Portfolio** | Position 집합 + 현금 + 일별 스냅샷 이력. 멀티 포트폴리오 지원 |
| **Rule** | 개별 조건 하나 (텍스트 → Python 코드로 변환) |
| **Strategy** | Rule들의 묶음. 필수/옵션 규칙 구분. 폴링 주기 및 활성화 여부 관리 |
| **Signal** | 전략이 생성하는 매매 지시 (BUY / SELL / HOLD) + 충족된 규칙 목록 |

## 8. 핵심 설계 원칙

1. **View 중심 구조**: 모든 기능은 Portfolio View를 기준으로 동작
2. **전략과 실행의 분리**: 전략 정의 ≠ 실행. 실행은 별도 엔진에서 처리
3. **규칙/전략 구조**: 규칙(Rule)을 만들고 전략(Strategy)으로 묶어 관리. 필수/옵션 규칙 구분
4. **접근성**: 코드를 몰라도 규칙 작성 가능. 아는 사람은 직접 수정 가능
5. **DataService 추상화**: localStorage → DB 교체 시 컴포넌트 무수정
6. **단계적 확장**: View → 전략 → 실행 → 자동화 → 외부 서비스
